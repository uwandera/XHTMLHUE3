1. O que é o XAMPP, em termos de arquitetura
XAMPP é essencialmente um pacote de distribuição integrada composto por quatro pilares:
X = Cross‑platform
Funciona em Windows, Linux e macOS.
A = Apache
Servidor web HTTP/S.
M = MySQL/MariaDB
Sistema de gerenciamento de banco relacional.
P = PHP
Linguagem de servidor embutida no Apache.
P = Perl
(hoje menos relevante, mas presente por herança histórica).
Ele funciona como um bundle de runtime, não como um ambiente de produção.
Seu foco é: desenvolvimento local, com tolerância a permissões fracas, portas livres e processamentos independentes.

🧩 2. Arquitetura Interna do XAMPP — Vista de sistema
Tecnicamente, o XAMPP roda como um gerenciador de serviços empacotados:
XAMPP Control Panel
│
├── Apache (httpd.exe ou bin/httpd)
│ ├── Módulo mod_php
│ ├── mod_ssl
│ ├── mod_rewrite
│ └── libphp.so/libphp.dll
│
├── MySQL/MariaDB (mysqld)
│ ├── Thread Scheduler
│ ├── Query Optimizer
│ ├── Armazenamento InnoDB
│ └── File I/O Manager
│
├── PHP (php.exe / php-cgi.exe)
│ ├── Zend Engine
│ ├── Opcode Handler
│ └── Extensões (pdo_mysql, curl, openssl)
│
└── Serviços acessórios
├── FileZilla FTP Server
├── Mercury Mail
└── Tomcat (Java servlet container)

O Control Panel age como um process manager, que:
Inicia processos nativos
Mantém logs em tempo real
Exibe portas em uso
Injeta variáveis de ambiente
Gerencia threads dos serviços
Ele não contém uma virtualização — tudo roda nativamente no sistema operacional.

🔥 3. Entendendo o Apache dentro do XAMPP
O Apache do XAMPP vem pré-configurado com:
🔹 Worker model
Na versão Windows, usa o módulo mpm_winnt, que cria:
1 processo pai
N threads por processo para tratar conexões
No Linux/macOS, pode usar mpm_prefork ou mpm_worker.
🔹 Pipeline de processamento de requisições
O fluxo:
Cliente → Listener Socket → Accept Thread → Worker Thread
→ mod_rewrite → mod_php → PHP Interpreter → Output Buffer → Cliente
🔹 mod_php
É crucial:
o PHP roda embutido no Apache, sem FPM.
Isso significa:
Sem pool de workers PHP
Sem gestão de processos PHP independentes
O PHP vive dentro da thread do Apache
Cada requisição exige carregamento de extensões PHP via Zend Engine
O resultado: simples de usar, não ideal para produção.

🐘 4. PHP dentro do XAMPP — como roda de verdade
O PHP no XAMPP opera em modo Apache SAPI, carregado via módulo:
LoadModule php_module modules/php7apache2_4.dll
Internamente:
🔹 Zend Engine
Interpreta o código PHP, passando por:
Lexing
Parsing
Compilation to OPcodes
Execution
Por exemplo, um echo "oi" vira algo como:
OP_PRINT "oi"
OP_RETURN
🔹 Extensões
O PHP do XAMPP vem com dezenas:
pdo_mysql
mysqli
openssl
curl
gd
mbstring
intl
Cada extensão é um módulo C carregado dinamicamente.
🔹 php.ini
O PHP configurado no XAMPP vem com:
display_errors = On (debug)
error_reporting = E_ALL
extension_dir mapeado para /ext/
upload_max_filesize e post_max_size ampliados

🧬 5. Banco de Dados — MariaDB/MySQL
XAMPP utiliza MariaDB, um fork otimizado do MySQL.
Arquitetura interna:
🔹 Thread-per-connection
Cada cliente cria uma thread dedicada.
🔹 InnoDB Engine
O motor padrão oferece:
MVCC (controle de concorrência multiversão)
Buffers de página
Journaling com redo logs
ACID completo
🔹 Arquivos internos
ibdata1 → tablespaces
ib_logfile0/1 → logs de redo
mysql/ → metadados do sistema
phpmyadmin/ → interface web

⚙️ 6. Como o XAMPP gerencia portas e processos
Apache:
Porta 80 / 443
Verifica se Skype, IIS ou outro serviço está ocupando
MariaDB:
Porta 3306
FileZilla:
Porta 21 + portas passivas
Mercury:
Portas 25, 110, 143 (SMTP/POP3/IMAP)
O XAMPP usa a DLL netstat embutida para detectar conflitos.

🛰️ 7. Segurança — por que XAMPP NÃO é para produção
Ele vem propositalmente inseguro:
Apache com permissões amplas
Diretórios com listagem habilitada
MariaDB sem senha local
Configurações de debug do PHP habilitadas
SSL configurado de forma simplificada
Motivo: foco total em desenvolvimento local.
